5) Explain how DMA works. What are some of the issues that you need to worry about when using DMA?
    - Direct memory access. Helps with transferring large chunks of data in parallel with CPU (improves performance). 
    - How it works:
        1. Initiate the DMA controller: initiate size, buffer pointers, etc.
        2. Transfer: DMA controller takes over and accesses the memory and peripheral devices. Reads data from the source location and writes to the destination or vice versa.
        3. Completion: Once complete, the DMA controller can generate an interrupt. Allows the CPU to resume control and process the transferred data or take actions.
    - Problems:
        1. Data coherency: since the CPU isn't involved, it might not be aware of changes. Can lead to inconsistencies if not properly synchronized.
        2. memory protection: DMA bypasses mem protection mechanisms of the CPU. Could lead to unauthorized access or modification of memory locations.
        3. interrupt handling problems.
        4. Arbitration: if multiple components need the DMA controller, arbitration/scheduling mechanisms need to be in place.
        5. alignment and size.
        6. buffer overflows/underruns if DMA is faster than source or dest can handle.
        7. Concurrancy/Synchronization: with multi processors/threads, synchronization mechs need to be in place to ensure consistency

13) What is a pull-up or pull-down resistor? When might you need to use them?

    - Fixes floating input problem (high impedence lines have undefined voltage when nothing is driving them)
    - Pull-up connects the signal to the supply voltage. Commonly used for:
        - Buttons
        - Open-drain/open-collector outputs like I2C
        - Reset pins, so the processor continues running unless a device actively pulls it low
    - Pull-down connects the signal to ground

Extras:
 - What is the stack used for?
    - used to facilitate the control flow of a program and manage the memory required for function calls, parameter passing, and temporary storage of data including"
        - Function Calls and Return Addresses
        - Local variables
        - Parameter Passing: parameters are often passed through the stack. The calling function pushes the parameters onto the stack before making the function call
        - Context Switching
        - Interrupt Handling
        - Temporary Data Storage
        - Nested Function Calls
        - Resource Allocation and Deallocation

 - Where is the stack located in the memory map?
    - Embedded Systems with Downward Stack Growth: starts at the highest memory address and grows downwards toward lower memory addresses. The stack pointer points to the top of the stack, and it moves downwards as items are pushed onto the stack.
    - Embedded Systems with Upward Stack Growth: starts at a lower memory address and grows upwards toward higher memory addresses. The stack pointer (SP) points to the bottom of the stack, and it moves upwards as items are pushed onto the stack
    - Others: the stack is located within a process's memory space. The operating system usually allocates a specific range of addresses within the process's address space for the stack
 
 - Where is the heap located?
    - Embedded: determined by the memory layout defined in the system's linker script or memory configuration. The heap can be placed in a specific region of memory, either before or after the stack
 
 - Where are static/global variables located?
    - memory is divided into multiple sections, such as text (code), data, and stack sections. In this case, the data section typically contains initialized global and static variables. Determined by linker script.
    
 - What does the linker script do?
    - a configuration file used by a linker during the compilation process to specify how the program's code and data should be organized in memory. Linker scripts provide instructions to the linker about memory layout, sections, addresses, and other attributes of the final executable.
 
 - What is the compilation process?
    - Preprocessor: handles/copy-and-pastes pre-processor directives
    - Compilation: translates the C code into assembly language or an intermediate representation specific to the target architecture. The output is one or more object files. Optimizes
    - Linking: combines all the object files generated in the previous steps and resolves references between them. Also organizes memory layout and prepares the executable binary.
    - Generate ouput files: Included with final executable. Debug files like .map or .pdb
    
 - How are negative numbers represented in binary?
    - Two's compliment: MSB indicates that the number is signed. The rest of the bits are inverted (ex. 10000010 = -101 (-126)). The "~" operator inverts bits.
 
 - What's are some differences between FLASH and RAM?
    - Flash is non-volatile and RAM is volatile. The contents of RAM memory are lost when power is turned off. Flash is slower read/write. Flash has limited write/erase cycles.
 
 - How do interrupts work?
    - Interrupts are facillitated by the Interrupt Vector Table. This maps interrupt sources to their service routines. The IVT is a table of function pointers. Process:
        1: An interrupt signal occurs, which informs the processor of an event.
        2: The processor saves the current context (registers, program counter, etc.) of the current task onto the stack.
        2.5: In some cases interrupts can be temporarily disabled to prevent nested interrupts that might interfere with the current ISR execution.
        3: The processor uses the interrupt number to index the interrupt vector table and locate the corresponding ISR function pointer.
        4: The processor jumps to the address pointer by the ISR function pointer and starts executing the ISR. 
        5: Once the ISR completes the processors restores the saved context from the stack.
        6: If any interrupts were disabled, they are now re-enabled.
        7: The processor returns to the interrupted task's executino point and continues running the program.
    - In addition to this process, the processor has mechanisms for prioritizing and handling nested interrupts.
    
 - What should you never do in an interrupt?
    - Block or delay: this could lead to missed or delayed handling of events. Mess with timing.
    - Use std lib functions: These functions can be non-deterministic in terms of execution time and memory usage which can disrupt timing constraints.
    - Modify shared data unsafely: modifying shared data without proper synchronization can lead to data corruption and race conditions.
    - Nested interrupts without care: 
    - Use floating point operations: floating-point operations can be significantly slower than integer operations and may not be suitable for ISRs where timing is critical.
    - Allocate dynamic memory: malloc can introduce unpredictable delays due to fragmentation and allocation overhead.
    - Use infinite while loops
    
 - What is "priority inversion"?
    - This is a phenomenon where a higher-priority task is temporarily blocked by a lower-priority task due to resource contention. This is a step-by-step explanation of how it can occur:
        1: There a multiple task priorities. For example, high (Task A), medium (Task B), and low (Task C).
        2: Task A and C need to access a shared resource, such as a hardware device or a critical section of code.
        3: Task C runs and takes the resource.
        4: Task C is preempted by Task B. Now Task A cannot run until Task C is complete and releases the resource.
 - How can you prevent priority inversion?
    - Priority inheritence: The lower-priority task is temporarily elevated to that of a high-priority task while it holds a shared resource.
    - Priority ceiling: When a task accesses a shared resource, its priority is temporarily elevated to the ceiling priority, preventing priority inversion.
    - Disabled interrupts when accessing a critical resource.
